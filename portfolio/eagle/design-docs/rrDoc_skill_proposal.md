# rrDoc 設計稿

> 📅 建立日期：2026-01-28
>
> **rrDoc** = **R**ead **R**eview Result + Update **Doc** 的縮寫
>
> 讀取 `/reviewDoc` 的檢核結果，根據改進建議更新 proposal，然後再次執行 `/reviewDoc`

---

## 問題描述

### 現況痛點

目前 `/reviewDoc` 檢核後的流程：

```
/reviewDoc → 列出修改建議 → 用戶手動更新文件 → /reviewDoc again
```

問題：
1. 用戶需要手動根據建議修改文件
2. 修改完還要記得執行 `/reviewDoc again`
3. 兩個步驟分開，容易遺漏

### 預期效益

- 一個指令完成「根據建議更新文件 + 再次檢核」
- 用戶只需看結果，決定是否再執行 `/rrDoc`
- 保留用戶控制權，不自動循環

---

## 完整工作流程

```
完整 Debug 流程
│
├─ 1. /debugP dev → 生出 proposal
│
├─ 2. /reviewDoc → 列出修改建議
│
├─ 3. 用戶看過沒問題
│
├─ 4. /rrDoc → 更新文件 + 再次 /reviewDoc
│
├─ 5. 用戶看結果
│   ├─ 還要改 → 再執行 /rrDoc
│   └─ 沒問題 → 結束（可執行 /implement）
│
└─ 6. 重複步驟 4-5 直到滿意
```

---

## 執行流程圖

```
/rrDoc 執行流程
│
├─ 1. 【取得上次檢核結果】
│   ├─ 從對話上下文找到最近一次 /reviewDoc 的結果
│   └─ 提取「發現的問題」區塊中的修改建議
│
├─ 2. 【更新文件】
│   ├─ 根據修改建議逐項更新提案文件
│   ├─ 記錄每項修改內容
│   └─ 區分修改類型：
│       ├─ 可直接修正 → 執行修正
│       └─ 需用戶確認 → 標註「⚠️ 待確認」
│
├─ 3. 【再次檢核】
│   └─ 執行 /reviewDoc again
│
└─ 4. 【輸出結果】
    ├─ 顯示本次修正摘要
    └─ 顯示 /reviewDoc 檢核結果
```

## 可直接修正的項目

| 類型 | 可直接修正 | 說明 |
|------|------------|------|
| DTO description 補充 | ✅ | 根據欄位名稱和類型補充 |
| DTO example 補充 | ✅ | 根據欄位類型生成範例 |
| 欄位一致性（create/update 同步） | ✅ | 在缺少的 endpoint 補上欄位 |
| 文件結構調整（區塊順序） | ✅ | 移動區塊到正確位置 |
| 欄位影響矩陣補充 | ✅ | 根據修改內容生成矩陣 |
| Service 層 select 補充 | ✅ | 補充遺漏的欄位到 select |
| 業務邏輯問題 | ⚠️ | 標註待確認，不自動修正 |
| 架構設計問題 | ⚠️ | 標註待確認，不自動修正 |

## 實作細節

### Skill 嵌套調用

`/rrDoc` 會在最後調用 `/reviewDoc again`：

```markdown
## 任務

1. 從上下文取得 /reviewDoc 的修改建議
2. 根據建議更新文件
3. 執行 /reviewDoc again  ← 嵌套調用
4. 輸出結果
```

### 如何取得上次 /reviewDoc 結果

從對話上下文搜尋最近一次 `/reviewDoc` 輸出中的：
- 「❌ 發現的問題」區塊
- 每個問題的「建議」內容

### 注意事項

1. **必須有前置 /reviewDoc**：如果上下文沒有 /reviewDoc 結果，提示用戶先執行
2. **保留用戶控制權**：不自動循環，用戶決定是否再次執行
3. **區分可修正 vs 待確認**：業務邏輯問題標註但不自動修正

```markdown
## 說明

**rrDoc** = **R**ead **R**eview Result + Update **Doc** 的縮寫

讀取前一次 `/reviewDoc` 的檢核結果，根據改進建議更新 proposal，然後自動執行 `/reviewDoc again`。

## 前置條件

必須先執行過 `/reviewDoc`，對話上下文中要有檢核結果。

## 執行流程

```
/rrDoc 執行流程
│
├─ 1. 【取得檢核結果】
│   └─ 從上下文找到 /reviewDoc 的修改建議
│
├─ 2. 【更新文件】
│   ├─ 可直接修正 → 執行修正
│   └─ 需用戶確認 → 標註「⚠️ 待確認」
│
├─ 3. 【再次檢核】
│   └─ 執行 /reviewDoc again
│
└─ 4. 【輸出結果】
    └─ 修正摘要 + 檢核結果
```

## 任務

1. **檢查前置條件**：確認對話上下文有 `/reviewDoc` 的檢核結果
   - 沒有 → 提示：「請先執行 `/reviewDoc` 進行檢核」
   - 有 → 繼續
2. **提取修改建議**：從「❌ 發現的問題」區塊提取每個問題和建議
3. **逐項更新文件**：
   - 可直接修正的項目（DTO 補充、結構調整等）→ 直接修正
   - 需用戶確認的項目（業務邏輯、架構設計）→ 標註「⚠️ 待確認」，不修正
4. **記錄修正內容**：建立修正摘要表格
5. **執行 /reviewDoc again**：嵌套調用 /reviewDoc 進行再次檢核
6. **輸出結果**：
   - 本次修正摘要
   - 已更新文件清單
   - /reviewDoc 檢核結果
   - 後續操作提示
```
