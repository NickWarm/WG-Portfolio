# /add-pi 執行流程

```
/add-pi 執行流程（v1.2 — 將 Data Flow 潛在問題納入 Proposal）
│
├─ 1. 【定位來源】
│   ├─ 從對話上下文取得 proposal 路徑
│   ├─ 讀取 proposal，搜尋「## API Data Flow 參照」
│   │   ├─ 有 → 提取 data-flow 文件路徑
│   │   └─ 沒有 → 從 proposal 推斷 apiName
│   │       ├─ 推斷規則：路徑含 {xxx}_api/ → apiName = {xxx}
│   │       ├─ ⚠️ 唯一搜尋目錄：prompts/6_api_data_flow/{adminApi|publicApi}/
│   │       ├─ 精確搜尋：{apiName}-data-flow.md（❌ 禁止 Glob 模糊搜尋）
│   │       ├─ 找到 → 繼續
│   │       └─ 沒找到 → 提示「請先執行 /api-flow-architecture」→ 結束
│   └─ 讀取 data-flow 文件
│
├─ 2. 【提取所有已知問題】
│   ├─ 「## 問題清單」→ 所有 ❌ 和 ⚠️ 問題（含各子表格）
│   ├─ 「## 500 防護檢查 → 500 路徑清單」→ ⚠️ 未防護路徑
│   ├─ 「## 資料流驗證 → 資料流斷點彙總」→ ❌ 和 ⚠️ 斷點
│   ├─ 「-c 檢查發現的問題」→ UI component 發現（找不到就跳過）
│   ├─ 「-s 檢查發現的問題」→ Situation 發現（找不到就跳過）
│   └─ 合併去重（問題描述 + 位置為唯一鍵，保留最高嚴重度）
│
├─ 2.5 【⚠️ 問題篩選】（v1.1 新增）
│   ├─ ❌ 問題 → 全部保留
│   └─ ⚠️ 問題 → 逐一判斷
│       ├─ 排除：前端已防護（catch/fallback/disabled）
│       ├─ 排除：無人使用（無前端頁面/前端未使用）
│       ├─ 排除：冗餘欄位（API 回傳但前端未使用）
│       └─ 保留：可能導致錯誤/500/資料不正確
│
├─ 3. 【比對 proposal 現有內容】
│   ├─ 掃描 proposal 的修改建議（解法、後端修改、前端修改）
│   ├─ 已涵蓋的 → 標記「✅ 已涵蓋於 § 解法 N」
│   └─ 未涵蓋的 → 標記「🆕 需新增」
│
├─ 4. 【寫入 proposal】
│   ├─ 全部已涵蓋 → 不新增區塊
│   │   └─ 顯示「✅ 已涵蓋所有已知問題」→ 結束
│   └─ 有未涵蓋 → 新增/覆蓋「## 已知潛在問題（Data Flow）」
│       ├─ 來源標註（data-flow 路徑）
│       ├─ ❌ 需修正：問題表格 + 每個問題的建議修改方式
│       ├─ 欄位名不匹配問題 → 依「欄位名不匹配統一修法規則」處理（後端加 alias，不改前端）
│       ├─ ⚠️ 注意事項：問題表格（只列表，不寫建議修改）
│       └─ 寫入位置：proposal 末尾（「## 修改檔案清單」之前）
│
└─ 5. 【輸出】
    ├─ 納入 N 個已知問題（M 個 ❌ + K 個 ⚠️）
    ├─ 其中 X 個已被 proposal 涵蓋
    └─ 提示：「請執行 /reviewDoc 統一檢核」
│
└─ 6. 【更新進度表】（@update-progress.md）
    └─ 將 /add-pi 標記為 ✅
```
